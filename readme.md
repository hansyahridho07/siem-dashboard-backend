# Backend Technical Test

## Overview

In this assessment, you are required to build a mini API service that functions as the backend for a Security Information and Event Management (SIEM) Dashboard.

The system combines:

- Internal asset information stored in PostgreSQL
- Security alert logs stored in Elasticsearch

The required infrastructure has already been provided through Docker Compose. Your task is to build the backend service that integrates and aggregates data from both sources.

---

## Evaluation Goals

This assessment is designed to evaluate:

- API design
- PostgreSQL integration
- Elasticsearch integration
- Data aggregation and enrichment
- Code organization and maintainability
- Error handling
- Technical decision making

Candidates should be prepared to explain their design decisions during the technical interview.

# Existing Data Sources

## PostgreSQL

### Table: internal_infrastructure_assets

Stores master information about internal company assets.

| Field                 | Type        |
| --------------------- | ----------- |
| id                    | SERIAL (PK) |
| asset_name            | VARCHAR     |
| host_identifier_local | VARCHAR     |
| department_owner      | VARCHAR     |
| risk_level            | VARCHAR     |

---

## Elasticsearch

### Index: security-alerts

Stores raw security alerts generated by security monitoring systems.

| Field             | Type    |
| ----------------- | ------- |
| timestamp         | date    |
| src_ip            | ip      |
| network_target_ip | ip      |
| signature_name    | keyword |
| severity          | integer |

# Tasks

## Task 1 – Advanced Alert Filtering

Create an endpoint that allows the frontend to filter security alert logs using asset information stored in PostgreSQL.

### Query Parameters

| Parameter  | Required |
| ---------- | -------- |
| department | Optional |
| risk       | Optional |
| page       | Optional |
| limit      | Optional |

### Expected Behavior

If a request filters by department = Finance, the API should return alert logs whose target IP address belongs to an asset owned by the Finance department.

Asset ownership information must be obtained from PostgreSQL and correlated with Elasticsearch alert data.

### Response Format

```json
{
  "success": true,
  "message": "Successfully fetched alert logs",
  "meta": {
    "total_data": 12,
    "page": 1,
    "limit": 20
  },
  "data": [
    {
      "timestamp": "2026-06-01T10:01:00Z",
      "source_ip": "185.220.101.5",
      "target_ip": "192.168.20.50",
      "alert_name": "BROWSER-CHROME CVE-2023-3079 Exploit Attempt",
      "severity": 1
    }
  ]
}
```

## Task 2 – Dashboard Aggregation & Enrichment

Create an endpoint used by dashboard widgets to display the most frequently targeted internal assets.

### Expected Behavior

Retrieve the Top 5 targeted IP addresses from Elasticsearch based on attack frequency.

Enrich the aggregated result using asset information from PostgreSQL.

The final response should include:

- Target IP
- Total attacks
- Asset name
- Department owner

### Response Format

```json
{
  "success": true,
  "message": "Successfully fetched top targeted assets",
  "meta": {
    "total_data": 5
  },
  "data": [
    {
      "target_ip": "192.168.10.25",
      "total_attacks": 6,
      "asset_name": "Server Core Finance",
      "department": "Finance"
    }
  ]
}
```

## Task 3 – Highlighted IP Monitoring

Security analysts need a mechanism to highlight suspicious external IP addresses and monitor their activity.

### Part A – Database Design

Create a new PostgreSQL table to store highlighted IP addresses.

The schema design is entirely up to you.

### Part B – CRUD API

Implement endpoints to:

- Add highlighted IP
- View highlighted IP list
- Update highlighted IP
- Delete highlighted IP

### Part C – Activity Monitoring

Create an endpoint that returns alert logs from Elasticsearch where:

```text
src_ip matches a highlighted IP stored in PostgreSQL
```

If there are no highlighted IPs, return an empty data array.

### Response Format

```json
{
  "success": true,
  "message": "Successfully retrieved data",
  "meta": {
    "total_data": 2
  },
  "data": []
}
```
